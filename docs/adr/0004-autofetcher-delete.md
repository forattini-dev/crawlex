# AutoFetcher is deleted; `policy::engine` remains the auto-escalation source of truth

`runner::AutoFetcher` was introduced in slice #21 (PRD #15) as a value-shaped type that composes `SpoofFetcher + RenderFetcher + ChallengeDetector` and signals escalation via `JobOutcome.retry`. It has never had a production consumer: the live auto-path escalation in `crawler.rs` flows through `policy::engine` consuming `ChallengeDetector::detect` directly, with the Crawler re-queueing on `Decision::Render`. Slice A4 of PRD #24 makes the choice explicit: **delete AutoFetcher**. The Crawler stays with `policy::engine` as the single arbiter of "this job needs to be re-queued as Render".

We considered the alternative of promoting AutoFetcher — wiring it into `JobRunner::run` for `Method::Auto`, thinning out the antibot escalation logic in `policy::engine`, and letting the runner's `RetryDecision::Suggest { EscalateToRender }` be the only escalation signal. Rejected because: (1) `policy::engine` already evaluates cross-job context the runner does not see — retry caps, host cooldowns, render budgets, proxy score floors — and the antibot decision is *one* input among several; pulling antibot escalation into the runner forces the runner to either consult the same cross-job state (defeats the boundary) or hand back a richer signal the Crawler still needs to interpret (no real simplification); (2) AutoFetcher has zero consumers on `main` today — promoting it means moving production code; deleting it means deleting test-locked dead code; (3) the policy/engine boundary is where retry vs render vs drop decisions belong (already documented in PRD #15 Q11). Antibot vendor identification (the bit that *triggers* escalation) lives in `fingerprint::target::sources::AntibotMarkerSource` after B5; `policy::engine` consumes it via `Fingerprinter::analyze_hot` after B14.

The consequence: `runner::AutoFetcher` and `runner::AutoOutcome` are marked `#[deprecated]` in this slice; they continue to compile and their unit tests continue to pass (proving the composition semantics for any future caller that needs the same shape) but no production code path constructs them. Final deletion happens in B15 alongside the `tech_fingerprint` / `block_detector` / `challenge.rs` cleanup. If a future requirement appears that genuinely needs a per-Job escalation arbiter — e.g., wanting `Method::Auto` to be configurable per-job rather than per-run via the policy engine — reopen this ADR and restore AutoFetcher rather than silently re-implementing it.
